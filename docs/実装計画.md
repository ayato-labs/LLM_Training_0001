# 実装計画書: LLM_Training リファクタリング（引き算のエンジニアリング）

## 1. 目的・背景
- **制約**: 個人開発・GPUリソース極小（VRAM 4GB〜24GB想定）・ローカルSSD容量逼迫
- **課題**: コードベースが肥大化し、学習実行までのオーバーヘッド（HPO実行、複雑な依存関係、独自実装の保守）がボトルネック化
- **方針**: 「エンジニアリングは引き算」
    - 独自実装 → 標準ライブラリ委譲
    - 実行時探索 → 事前探索（アーティファクト化）
    - 複数ログ基盤 → 単一ログ基盤
    - ただし **Google Drive自動アップロード（SSD節約）は維持**

## 3. 設定管理の新アーキテクチャ（Hydra `defaults` 活用）

### 3.1 ファイル構成
```text
configs/
├── config.yaml          # ベース設定（共通・変更頻度低）
├── hparams_150M.yaml    # 探索済み最適HP（モデルサイズごとに作成）
└── hparams_3B.yaml      # 将来拡張用
```

### 3.2 `config.yaml` の `defaults` 記述例
```yaml
defaults:
  - config       # config         # ベース設定
  - hparams_150M # モデルサイズ固有の最適HP（自動生成される）
# 実行時上書き: python -m src.main +experiment=debug max_steps=100
```

### 3.3 `hparams_150M.yaml` 記述例（生成物）
```yaml
# @package _global_
# Generated by scripts/find_hparams.py
training:
  max_lr_2d: 2.8e-4
  max_lr_1d: 2.5e-3
  batch_size_seqs: 16
  warmup_ratio: 0.035
  weight_decay: 0.1
  beta2: 0.95
  grad_clip: 1.0
```

## 4. 学習パイプラインの簡素化フロー（`main.py` 目標実装）

```mermaid
flowchart TD
    A[CLI引数解析] --> B[Hydra設定読込\nconfig + hparams_XXX]
    B --> C[Seed固定\nset_seed]
    C --> D[Tokenizer読込]
    D --> E[Dataset読込・Tokenize\nDataCollator]
    E --> F[Model初期化\nLlamaForCausalLM]
    F --> G[DeepSpeed設定読込\n静的JSON or HF自動]
    G --> H[TrainingArguments構築]
    H --> I[Trainer構築\n+ DriveUploadCallback]
    I --> J[train()]
    J --> K[Model/Tokenizer保存]
    K --> L[Drive最終アップロード]
```

**削除されるフロー**: Optuna Study作成 → Proxy学習ループ → BestParams適用 → Step Law適用 → Pilot Check

## 6. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| `LlamaForCausalLM` への移行で学習曲線が変わる | 高 | 学習初期のLoss値を既存ログと比較。`hidden_size` が `num_heads` の倍数になるよう `config.py` で調整ロジック維持 |
| DeepSpeed 設定（ZeRO Stage）が自動判定で合わなくなる | 中 | `train_model.py` の `generate_deepspeed_config` を `main.py` に移植し、VRAM検出ロジックは維持 |
| Drive API 認証が切れて学習停止 | 中 | `drive_uploader.py` にサービスアカウント/リフレッシュトークン永続化ロジックを実装済みか確認。`try-except` で学習本体を止めない |
| Hydra `defaults` のマージ順序で意図せぬ上書き発生 | 低 | `@package _global_` 指定と、CLIでの上書き (`+key=val`) でテスト |
