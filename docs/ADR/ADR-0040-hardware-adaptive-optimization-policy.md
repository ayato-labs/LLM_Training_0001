# ADR-0040-hardware-adaptive-optimization-policy: ローエンド単一GPUにおけるハードウェア適応型最適化ポリシーの定義

- **Status:** Accepted
- **Date:** 2026-07-19
- **Deciders:** Ayato-labs (ayato-labs)

## Context

WSL2 (Linux) 環境への本番移行 (ADR-0039) に伴い、`torch.compile`（コンパイラ）や `liger-kernel`（Triton製融合カーネル）を有効化して事前学習の高速化を試みた。
しかし、ローエンド・エントリークラスの単一GPU（RTX 3050 Laptop, 物理VRAM 4.0GB）においてこれらすべての機能をONに設定したところ、想定速度（約5〜6s/it）を遥かに下回る **43s/it（約7〜8倍の減速）** という極端なパフォーマンス劣化と、初期ステップ終了直後の **Loss爆発（NaN発散）** が発生した。

調査の結果、既存の高速化ライブラリが想定している「マルチGPUおよびハイエンドGPU（A100/H100/RTX 4090等）」向けの最適化設計が、小規模GPUのハードウェア制約と激しく衝突し、以下の深刻なペナルティを発生させていることが判明した：

1. **WDDM（Windows Display Driver Model）のサイレント・スワップ（Paging）問題**:
   - `torch.compile` はカーネル融合を行うが、コンパイル初期（最初の数ステップ）にアロケータや中間グラフ生成のためにVRAMを瞬間的に大量消費する。
   - 物理VRAM（4.0GB）が極限状態でこれが起きると、WindowsのWDDMドライバーが物理VRAMから溢れたデータをメインメモリ（システムRAM）へ自動で退避（Paging）させる。
   - WDDMは、コンパイルが完了しVRAM要求が落ち着いた後も、一度メインメモリに落としたデータを自動的にVRAMへ書き戻さないため、以降すべてのステップが低速なPCIeバス経由での通信を強いられ、全体のステップ速度が恒久的にスローダウンする。
2. **Liger Kernelとコンパイラの競合（グラフブレイク）**:
   - Liger Kernelがデフォルトで `model.forward` を丸ごと置き換えるFLCE（Fused Linear Cross Entropy）は、Triton製の `autograd.Function` をラップしており、`torch.compile` のDynamoトレース時に深刻なグラフブレイク（Graph Break）を連発させる。これにより処理が細切れのEagerフォールバックとなり、コンパイルのオーバーヘッドだけを払う状態になる。
   - また、小規模GPUではレジスタファイル容量の小ささから、融合処理によるレジスタ圧迫が発生し、かえってVRAMローカル領域への退避（レジスタ・スピル）が発生して遅くなる。
3. **決定論的アルゴリズム（`deterministic=True`）のペナルティ**:
   - `set_seed()` による再現性確保のために決定論的アルゴリズムを強制すると、cuBLASおよびTritonの非決定論的高速カーネルがすべてブロックされ、ハードウェア能力が大幅に抑制される（10〜20%の減速）。
4. **WSL2マウントドライブ（9Pプロトコル）のI/Oボトルネック**:
   - Windows領域（`/mnt/c/`）に置かれた大容量データをシンボリックリンク等でWSL2内から読み出すと、ネットワークマウントと同等の遅延が発生し、GPUのデータ空き待ちが発生する。

---

## Decision

単一かつリソース制約の厳しいローエンドGPU（VRAM ≤ 6GB、RTX 3050 Laptopなど）における実用的な学習効率を最大化するため、ハイエンド前提の最適化を自動（または手動）でダウングレードする **「ハードウェア適応型最適化ポリシー」** を定義・採用する。

### 決定事項：

1. **`torch_compile` の動的無効化**:
   - VRAM容量が極めて少ない環境（VRAM ≤ 6GB）においては、WDDMによるメインメモリ・スワップを防止するため、`torch_compile: false`（無効化）を標準設定とし、Eagerモードで動かす。
2. **`use_liger_kernel` のトレードオフ判断**:
   - VRAM空き容量に余裕がある状態（現在の150Mモデル等、VRAM消費が2GB以下）では `use_liger_kernel: false` を標準とし、標準PyTorchの極めて高速なC++/CUDAネイティブカーネルを優先する。
   - 逆にモデルやバッチサイズを拡大し、OOM（Out of Memory）を回避しなければならない状態に達した時のみ、`use_liger_kernel: true` によるVRAM削減効果を利用する。
3. **再現性制約（`deterministic`）のコンフィグ化**:
   - シード値設定時の決定論的アルゴリズムの強制（`deterministic=True`）を排除し、`config.yaml` 内で `deterministic: false` を規定値として変更できるように変更する。開発フェーズや通常の学習ではパフォーマンスを最優先する。
4. **`/mnt/c/`（Windows領域）からのデータアクセスの厳禁**:
   - WSL2上のデータパス（`data_path`）は、Windows領域のマウントやシンボリックリンクによる参照（`/mnt/c/`）を完全に禁止する。
   - データセット（`dataset.jsonl`）は必ずWSL2ネイティブ領域（ext4、例：`~/` 配下）に実体ファイルとして直接コピー・配置して利用することを義務付ける。
5. **DataLoader並列度のCPU自動マッピング**:
   - GPUのデータ空き待ちを防ぐため、Linux環境下では `os.cpu_count()` に基づいて並列ローダー（`dataloader_num_workers`）を自動設定（コア数の25〜50%、上限4）する。

## Consequences

- **学習速度の劇的な改善**: 
  - `torch_compile` を無効化し、データの実体をWSL2ネイティブ領域へコピーした結果、スワップとI/Oボトルネックが共に解消され、ステップ速度は **43.0s/it から 7.01s/it へと約6倍高速化** された。
  - さらに `deterministic: false` によるカーネル最適化解放により、数%〜十数%の追加高速化が得られる。
- **学習の安定化**:
  - `warmup_ratio` 解決バグの修正により、コサイン学習率が初期の不安定なウェイトに適正に立ち上がるようになり、NaN発散（Loss爆発）が完全に排除された。
- **保守の容易化**:
  - ハードウェアスペックに起因するスワップ発生時、コンパイル後のピークアロケータメモリ量（`peak_reserved`）を監視してシステム警告ログを自動発報する仕組み（`DetailedLoggingCallback`）が構築され、目に見えないパフォーマンス劣化を即座に検知・切り分け可能になった。
