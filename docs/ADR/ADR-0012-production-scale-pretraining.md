# ADR-0012: 本格的な事前学習（ステップ数の拡張・エポックベースへの移行）とチェックポイント保存・容量管理ポリシーの確立

## ステータス
Accepted

## コンテキスト

RTX 3050 Laptop (VRAM 4GB) 環境下でのインフラ検証（WDDMページングの解消、1ステップあたり約8.9秒への高速化、DeepSpeedのアクティベーションCPUオフロード導入）が成功し、50ステップのテスト学習が完了した。

しかし、テスト学習完了後のモデルによる生成テストでは、日本語の文法や語彙の接続を再現できず、単語や助詞が崩壊した断片が出力されるに留まった（未学習状態）。これはゼロからのフルスクラッチ事前学習において50ステップという学習量が圧倒的に不足していることによる正常な動作である。

モデルに実用的な作風表現力や日本語の言語構造を定着させるためには、事前学習を本格的なスケール（エポックベース）へ移行させ、長時間の学習を安定して完遂するインフラ戦略を定義する必要がある。

### 課題と前提条件
- **学習量の設計**:
  前処理済みのデータセットは約43,391チャンク（約4,400万トークン相当）。物理バッチサイズ=1、勾配累積=10（実効バッチサイズ10）の場合、1エポックあたり約4,339ステップが必要となる。
  言語モデリングの基礎的な収束を得るためには最低3エポック（約13,000ステップ）の学習が推奨され、現在のスループット（約9秒/step）換算で **約33時間** の連続稼働が必要となる。
- **ディスク容量の制限**:
  現在確保できているSSDの空き容量は約13.7GB。120Mモデルのチェックポイントは、オプティマイザ状態や勾配バッファを含むと1回あたり約1.8GB（30M Proxyモデルで約270MB）のサイズを占有するため、長期学習で発生するチェックポイントをすべて蓄積すると確実に容量不足（os error 112）でクラッシュする。

---

## 意思決定

本格学習への移行に伴い、以下の構成変更を決定・実施した。

1. **エポックベース学習への切り替え (`MAX_STEPS = -1`, `NUM_EPOCHS = 3`)**
   - ステップ制限を解除し、`MAX_STEPS = -1` に設定。
   - 本格学習のためのエポック数として `NUM_EPOCHS = 3` を新設・適用する。実効バッチサイズ10において、約13,000ステップ（約33時間の稼働）のフルスケール事前学習に移行する。
2. **ステップベースのチェックポイント保存戦略の定義 (`save_strategy="steps"`, `save_steps=1000`)**
   - 長期学習中の予期せぬ電源切断、OSアップデート、またはハードウェアクラッシュに備え、1000ステップ（約2.5時間相当の進捗）ごとに中間チェックポイントを自動保存する。
3. **容量パンク防止のための保存上限設定 (`save_total_limit=2`)**
   - ディスクの空き容量を圧迫しないよう、保持するチェックポイントの数を最大 `2` 個に制限する。新しいチェックポイントが保存された際、最も古いものは自動的に削除される。これにより、オプティマイザ状態を含めてもディスク占有量を最大約3.6GB以内に抑え込み、13.7GBの空き容量で安全に完遂させる。

---

## 結果と影響

### 1. 期待される効果
- **日本語モデリング能力の獲得**:
  3エポック（約13,000ステップ）の事前学習が実行され、初期の文脈や小説特有の語彙のつながり、プレフィックス（会話率や感情）に応じたトーンの制御ロジックがモデルに定着し始める。
- **レジューム可能性とディスク安全性の両立**:
  1000ステップごとの保存により、万が一途中で異常終了した場合でも、最大2.5時間の損失で学習を再開（`resume_from_checkpoint=True`）できるようになる。また、`save_total_limit=2` の動作により、33時間の学習中も容量超過によるクラッシュは一切発生しない。

### 2. 将来的な懸念と再検討条件
- 33時間の連続高負荷により、ノートPC環境（RTX 3050 Laptop）の熱暴走やバッテリー保護によるクロック低下（サーマルスロットリング）が発生する可能性がある。その場合は、途中で一時停止する運用（タスクのサスペンド）や、別の仮想GPUサーバーへの移行を検討する。
