# ADR-0012: 本格的な事前学習（ステップ数の拡張・エポックベースへの移行）とチェックポイント保存・容量管理ポリシーの確立

## ステータス
Accepted

## コンテキスト

RTX 3050 Laptop (VRAM 4GB) 環境下でのインフラ検証において、当初の設計であった「120Mモデル＋コンテキスト長1024」を実行した際、アテンション行列による動的メモリ消費が物理VRAM（実質空き約3.0GB）を超過し、WindowsのWDDM共有メモリ退避（ページング）により1ステップあたり約150秒（3エポックで558時間）まで劇的に速度低下する現象を検知した。

このため、GPUオンリーの超高速スループット（1ステップ約9秒前後）を維持しつつモデルの表現力を最大化するため、**「コンテキスト長を512に制限し、浮いたメモリを用いてモデルパラメータ数を150M（1.5億）までスケールアップする」** というトレードオフ設計を採用する。

しかし、テスト学習完了後のモデルによる生成テストでは、日本語の文法や語彙の接続を再現できず、単語や助詞が崩壊した断片が出力されるに留まった（未学習状態）。これはゼロからのフルスクラッチ事前学習において50ステップという学習量が圧倒的に不足していることによる正常な動作である。

モデルに実用的な作風表現力や日本語の言語構造を定着させるためには、事前学習を本格的なスケール（エポックベース）へ移行させ、長時間の学習を安定して完遂するインフラ戦略を定義する必要がある。

### 課題と前提条件
- **学習量の設計**:
  前処理済みのデータセットは約43,391チャンク（コンテキスト長512に分割したため、実際には約88,000チャンク・約4,400万トークン相当）。物理バッチサイズ=1、勾配累積=10（実効バッチサイズ10）の場合、1エポックあたり約8,800ステップが必要となる。
  言語モデリングの基礎的な収束を得るためには最低3エポック（約26,000ステップ）の学習が必要となり、現在のスループット（約9秒/step）換算で **約66時間** の連続稼働が必要となる。
- **ディスク容量の管理**:
  150Mモデルのチェックポイントは、オプティマイザ状態や勾配バッファを含むと1回あたり約2.2GB（30M Proxyモデルで約270MB）のサイズを占有するが、このストレージ制約は外部的手段（ユーザーによるディスク容量の物理的確保・別ドライブの利用）によって解決する。そのため、プログラム側のロジックでチェックポイント数を制限する必要はなく、過去のチェックポイントをすべて保持して学習軌跡を完全に追跡可能とする。

## 意思決定

本格学習への移行に伴い、以下の構成変更を決定・実施した。

1. **エポックベース学習への切り替え (`MAX_STEPS = -1`, `NUM_EPOCHS = 3`)**
   - ステップ制限を解除し、`MAX_STEPS = -1` に設定。
   - 本格学習のためのエポック数として `NUM_EPOCHS = 3` を新設・適用する。実効バッチサイズ10において、約26,000ステップ（約66時間の稼働）のフルスケール事前学習に移行する。
2. **モデル・コンテキスト構造のチューニング (`SEQ_LEN = 512`, `TARGET_PARAMS = 150_000_000`)**
   - コンテキスト長を `512` に削減することで、アテンション演算のメモリピークを4分の1に減少し、WDDMページングを完全に回避。
   - 削減したメモリをモデルサイズに割り振り、ターゲットモデルを **150M**（L=20, H=832, Heads=13など）に拡張。4GB VRAMでの表現力を最大化。
3. **ステップベースのチェックポイント保存戦略の定義 (`save_strategy="steps"`, `save_steps=1000`)**
   - 長期学習中の予期せぬ電源切断、OSアップデート、またはハードウェアクラッシュに備え、1000ステップごとに中間チェックポイントを自動保存する。
4. **保存上限の排除（外部容量確保前提）**
   - ディスク容量は外部で十分に確保されるため、`save_total_limit` などのロジック制約はあえて適用しない。これにより、学習中の全中間モデルを保持し、過去の任意のステップへの巻き戻しや推論評価を可能とする。
5. **新規学習とレジューム（途中再開）のバッチエントリーポイント分離設計**
   - `run_training.bat` （新規学習用：前処理 ＋ HPO探索 ＋ 本番学習開始）とは別に、`resume_training.bat` （再開用）を新設。
   - `main.py` に `--resume` コマンドライン引数を実装し、これが有効な場合は前回の設定 `current_run_config.json` に `"resume": true` を自動注入し、15分かかるProxy探索プロセスを完全にスキップして直接最終学習の `train_model.py` をレジュームモードで起動するバイパス設計を採用。
6. **監視アップローダーの再圧縮防止機構 (`.uploaded` フラグファイル) の導入**
   - ローカルに維持される最新チェックポイントが、30秒ごとのポーリングごとに無限に再圧縮・アップロードスキップを繰り返す高負荷ループを防止するため、アップロード（またはスキップ確認）完了時にフォルダ内に `.uploaded` フラグファイルを生成し、次回以降の圧縮処理をスキップするガードを実装。

## 結果と影響

### 1. 期待される効果
- **表現力と高速性の両立**:
  150Mモデルの表現力をGPU内に収めつつ、WDDMページングを完全に排除して 9秒/step の超高速学習が可能になる。
- **無停止性と運用ミスの防止**:
  新規学習と途中再開が別バッチファイルに分かれたため、誤操作による学習済データの損失を防ぎ、停電や強制OSアップデート後も `resume_training.bat` のワンプッシュで即時に過去のオプティマイザの慣性（モーメンタム等）を100%保持したまま数学的に完全な学習復帰が可能。
- **全学習軌跡の保持とレジューム性能**:
  1000ステップごとの保存により、万が一途中で異常終了した場合でも、最大2.5時間の損失で学習を再開（`resume_from_checkpoint=True`）できるようになる。また、全チェックポイントが保持されるため、学習の途中でLossがどのように推移し、出力がいつ崩壊・改善したかを後からステップ単位で詳細に遡及分析できる。

### 2. 将来的な懸念と再検討条件
- 66時間の連続高負荷により、ノートPC環境（RTX 3050 Laptop）の熱暴走やバッテリー保護によるクロック低下（サーマルスロットリング）が発生する可能性がある。その場合は、途中で一時停止する運用（タスクのサスペンド）や、別の仮想GPUサーバーへの移行を検討する。
