# ADR-023: loguru 導入によるログ管理の統一

- **Status:** Accepted
- **Date:** 2026-07-11
- **Deciders:** Novel LLM Team

## Context

以前のログ管理は `print()` と `logging` モジュールが混在していた。

### 問題点

1. **ログレベルの非一貫性**
   - `print()` は常時出力 → デバッグ時と本番時で出力が変わらない
   - `logging` は設定が複雑 → 統一されたフォーマットが難しい
2. **ログの構造化不足**
   - `print()` は生テキスト → 機械的なパースが困難
   - JSON 形式でログを出力したいが、手動で実装が必要
3. **タイムスタンプの不一致**
   - `print()` はタイムスタンプなし
   - `logging` はフォーマットが自由だが、統一が難しい

## Decision

`loguru` を導入し、全ファイルの `print()` を `logger` に置き換える。

### loguru の特徴

1. **自動フォーマット**
   - タイムスタンプ、ログレベル、ファイル名、行番号を自動付与
   - デフォルトでカラフルなコンソール出力
2. **JSON 出力**
   - `sink` をファイルに指定するだけで、JSON形式でログを保存可能
   - 機械的なパースが容易
3. **簡単な設定**
   - `logger.add()` で追加のシンクを定義
   - ログローテーション、圧縮、保持期間を設定可能

### 実装方針

1. `src/logger.py` で loguru を設定
2. 全ファイルの `print()` を `logger.info()` に置き換え
3. 例外追跡: `try/except` + `logger.exception()` でスタックトレースを記録
4. ログ出力先:
   - コンソール: INFO レベル以上 (デフォルト)
   - ファイル: DEBUG レベル以上 (JSON 形式)

### ログファイルの構成

```
logs/
├── app.json          # 全ログ (JSON 形式)
└── run_20260711.log  # 実行ごとのログ (テキスト形式)
```

## Consequences

### メリット
- ログの一貫性: 全ファイルで同じフォーマット
- JSON 出力: 機械的なパースが容易
- タイムスタンプ自動付与: 手動追加不要
- デバッグ容易性: `logger.debug()` で詳細な情報を出力

### デメリット
- loguru の学習コスト (低い)
- ログファイルのサイズ増加 (ログローテーションで対応)

### 影響範囲
- `src/logger.py`: 新規作成
- `src/*.py`: 全ファイルの `print()` → `logger` 置き換え
- `pyproject.toml`: loguru を dependencies に追加
