# ADR-0010: Google Colab 統合の断念と専用 GPU クラウド（貸GPU）への移行
- 日付: 2026-06-28
- ステータス: 承認

## 背景
ローカルの `.py` モジュール構成を維持したまま Google Colab の無料/有料 GPU 計算資源を利用するため、バッチファイルによる同期や、AST解析を用いた単一 `.ipynb` ノートブック（インライン化）への変換ツールなどの統合環境を開発・検証した。
しかし、以下の課題により環境維持と開発のオーバーヘッド（摩擦）が極めて大きくなることが判明した。

1. **同期のオーバーヘッド**: ローカルでのコード修正のたびに、同期待ちやノートブックの再生成・再アップロードが発生し、開発サイクルが遅延する。
2. **セッションとストレージの制限**: Colab 側のセッション切れ、大容量ファイル（モデルやチェックポイント等）による Colab VM キャッシュ（Cドライブ容量）の枯渇問題など、実行環境の安定性に難がある。
3. **パスと環境の差分**: ローカル（Windows）と Colab（Linux）の環境差分（絶対パス、改行コード等）を吸収するためのパッチ処理やフォールバックコードが多数必要になり、本質的でないコードの複雑性が増大する。

## 決定
Google Colab を用いた `.py` プロジェクトの統合および学習の実行方針を**完全に断念**する。
今後は、RunPod や Vast.ai などの専用 GPU クラウド（貸GPU）から仮想マシン（VM）インスタンスをレンタルし、VS Code の **Remote-SSH 拡張機能** を利用して直接そのマシン上で開発・学習を行う方針へ移行する。

## 理由
1. **開発摩擦の極小化**: Remote-SSH を利用することで、ローカルで記述したコードが即座にリモート環境で反映され、パスのズレや同期の待ち時間が一切発生しなくなる（開発体験がローカルと同一になる）。
2. **本質的な開発への集中**: 環境差異のデバッグやファイルシステム上限の回避策に開発時間を費やすのは、モデル品質向上において非常に非効率であるため。

## 補足 (既存OSS調査結果)
調査の結果、Jupytext や py2nb、ipynb-py-convert などの既存OSSツールは存在するが、これらは「1つの `.py` ファイルを1つの `.ipynb` に変換する」または「両者を同期する」用途に留まる。
「プロジェクト内の複数ファイルのインポート依存ツリー（AST）を再帰的に解析し、単一の `.ipynb` ファイルへ自動でバンドル（%%writefile等による再現）する」という高度な機能を持った既存OSSは存在せず、自作ツールの保守コストがかかり続けることも断念の判断材料となった。
