# ADR-0042-offline-compilation-cache-warmup: オフライン・コンパイルキャッシュのウォームアップによるVRAMスパイクとスワップの回避- Liger + torch.compile は 依然として相性が悪い

- **Status:** Accepted
- **Date:** 2026-07-19
- **Deciders:** Ayato-labs (ayato-labs)

## Context

ローエンドGPU（RTX 3050 Laptop, 4GB VRAM）において `torch.compile`（コンパイラ）を有効化すると、最初のステップ実行時に発生するコンパイラおよびアロケータの瞬間的なVRAM消費スパイクが物理限界を超え、WindowsのWDDMドライバーによるシステムメモリ（RAM）への恒久的なスワップ（Paging）を誘発し、計算速度が極端に低下する問題が発生した（ADR-0040）。

この問題は、学習本番プロセスの中で「巨大なモデル・オプティマイザ状態」と「コンパイラのオートチューニング用ワークスペース」が同時に同じVRAM空間を競合することに起因する。

この競合を回避するため、本番学習開始前に、オプティマイザや余分なメモリ状態をロードしない軽量な状態で、実データ（最初の1バッチ）を流してコンパイルとオートチューニングを完了させ、その結果をディスク（`TORCHINDUCTOR_CACHE_DIR`）に固定化（キャッシュ）してから本番学習を開始する「時間軸での分離（オフライン・コンパイル）」という手法の有効性が浮上した。

---

## Decision

`torch.compile` 使用時の一時的VRAMスパイクとOSによるメモリ退避（スワップ）を防ぐため、本番学習プロセスの起動前にコンパイルキャッシュを焼き付ける **「オフライン・コンパイルキャッシュのウォームアップ（Warmup）フェーズ」を導入する。**

### 具体的な設計および実装方針：

1. **実データを用いたウォームアップの実行**:
   - ダミーデータによるテンソル形状（Shape）の不一致に伴う本番学習時の「再コンパイル（Recompiles）」を防ぐため、実際のデータセットおよびデータローダーから取得した「最初の1バッチ（実データ）」を用いてウォームアップを実行する。
2. **メモリ競合の分離（時間軸の分離）**:
   - ウォームアップスクリプト（または学習のウォームアップ段階）では、オプティマイザ（AdamW）やモデル勾配、余分なコールバックなどを一切構築せず、純粋に「モデルのフォワードおよびバックワード」のコンパイルのみを実行する。
   - これにより、コンパイル時の一時ワークスペース（オートチューニング領域）が4.0GBのVRAM上限に衝突することを防ぐ。
3. **`TORCHINDUCTOR_CACHE_DIR` による永続化**:
   - 環境変数 `TORCHINDUCTOR_CACHE_DIR` をプロジェクトディレクトリ内の特定のパス（例：`models/output/.inductor_cache`）に固定し、ウォームアップによって生成された最適化済み Triton カーネルをディスクに永続キャッシュ化する。
   - 本番学習実行時には、このキャッシュディレクトリを読み込ませることで、実行時の動的コンパイルおよびVRAMスパイクを完全にゼロにする。

---

## Consequences

- **メリット**:
  - 4GB VRAM環境であっても、本番学習時にコンパイル時のVRAMスパイクが発生しないため、OSによるサイレント・スワップを完全に回避しつつ、`torch.compile` 本来の計算効率（速度向上）の恩恵を受けられる可能性がある。
  - テンソル形状が完全一致する実データを用いるため、実行時に再コンパイルが走るリスクが極めて低い。
- **トレードオフと追加作業**:
  - 本番学習を開始する前に、最初の1回だけ数分間の「キャッシュ焼き込みプロセス」を走らせる必要があり、起動フローが2フェーズ化する。
  - 実装検証において、キャッシュ読み込みが本当に本番プロセスでヒットし、VRAMスパイクが発生しないかを確認する監視・ログ出力ロジックが必要。
