# 概念的要件定義書

**プロジェクト**: LLM_Training (個人GPU環境におけるスクラッチ事前学習パイプライン)  
**バージョン**: 2.0  
**ステータス**: 基準文書（実装および設計判断の最終決定ポリシー集）

---

## 1. 根本原則：引き算のエンジニアリング

> **「コードを書くことは借金である。エンジニアリングは引き算をする職人仕事である。」**

本プロジェクトにおけるすべての技術・アーキテクチャの選定は、以下のポリシーに沿って判断される。

### 1.1 開発優先順位ポリシー
1. **書かない**（既存の標準ライブラリ・上流フレームワークに委譲する方針）
2. **削除する**（独自コードや不要になったデッドコード、重複機能を積極的に廃止する方針）
3. **分離する**（実行時計算を減らし、静的・オフラインな事前計算成果物に切り出す方針）
4. **統一する**（設定やデータ形式などの二重管理を排除し、単一ソース化する方針）
5. **最小限の接着コードのみ書く**（フレームワークの連携や動作上の制約回避に必要なもの以外は実装しない方針）

## 2. アーキテクチャ憲章（技術スタック選定ポリシー）

### 2.1 採用する標準スタック（Adopt ポリシー）

| 領域 | 採用するスタック・技術 | 採用の決定方針・理由 |
|---|---|---|
| **モデル実装** | `transformers.LlamaForCausalLM` + `default_data_collator` | Llama構造の標準APIに準拠し、拡張性と安定性を最大化するため採用。 |
| **学習ループ** | `transformers.Trainer` | 勾配累積、AMP、チェックポイント管理などの成熟した学習ループの保守コストを排除するため全面採用。 |
| **オプティマイザ** | `AdamW` (8-bit / Paged bnb) | ローエンドGPU環境におけるオプティマイザステートのVRAM消費（約75%）を極小化するため採用。 |
| **設定管理** | `Hydra` + `OmegaConf` | コマンドラインからの動的オーバーライドと静的YAMLスキーマの合成を統一的に行うため採用。 |
| **ログ基盤** | `TensorBoard` | サードパーティ製のクラウドSaaSやローカルWebUI（MLflow等）の維持コストを排除するため、最小限のローカル可視化として採用。 |
| **データ処理** | `datasets` + オンザフライパッキング | パディングを完全に排除したシーケンスを構成し、CPU/メモリマップ読み込みを最大化するため採用。 |
| **分散・メモリ** | `DeepSpeed` | 単一GPUにおけるメモリ節約（ZeRO）とCPUオフロードをライブラリ側に完全委譲するため採用。 |
| **環境管理** | `uv` | 環境構築の高速化と、依存パッケージ（OS毎のプラットフォームマーカー適用を含む）の確実なロック管理のため採用。 |

### 2.2 明示的に採用しないもの（Avoid ポリシー）

| 領域 | 採用しないスタック・アプローチ | 不採用の決定方針・理由 |
|---|---|---|
| **独自モデル実装** | カスタムTransformer、独自アテンション等 | 標準モデルで必要十分。独自アテンションやカスタマイズの追加・保守は重大な技術負債となるため不採用。 |
| **独自Trainer** | 手動の学習ループ、カスタムトレーニングコード | 独自ループの作成は不具合の温床であり、フレームワークの追従コストを支払う価値がないため不採用。 |
| **実験管理ツール** | `MLflow`、`Wandb`、DVCアーティファクトストア等 | ローカル個人開発における実行依存とローカルディスクの二重消費を回避するため不採用。ログはTensorBoardに一本化。 |
| **実行時最適化** | 学習開始時（ランタイム）のHPO探索・Optuna実行 | HPOによる試行オーバーヘッド（数分〜数時間）を学習ループに持ち込ませないため、実行時探索は一切不採用。 |
| **設定ファイルの乱立** | `scaling_*.yaml` などのベース複製ファイル群 | 変更差分を複数のファイルで二重管理することを避けるため、設定は単一ベースと hparams の合成パターンに制限。 |
| **ネットワーク依存** | Hugging Face Hub 経由の直接トークナイザロード | ネットワーク切断、レートリミット、および認証エラーによるパイプライン停止を防止するため、ローカルファイル直接読み込み以外不採用。 |
| **自動DeepSpeed構成** | DeepSpeed構成JSONの静的なハードコーディング | ハードウェアの空きVRAMに応じて動的にZeRO設定（CPUオフロード有無など）を生成する方針をとるため、静的YAML/JSONの管理は不採用。 |

## 3. フェーズ分離ポリシー

本プロジェクトは、計算および探索オーバーヘッドを学習ランタイムに侵入させないため、フェーズを空間的・時間的に完全分離する。

### 3.1 探索フェーズ（オフライン・成果物指向）
- **ポリシー**: ハイパーパラメータ探索（Optuna）や Step Law に基づくモデルサイズ・学習率スケーリング計算は、学習実行前に完全に独立したフェーズとして実行する。
- **制限事項**: 学習用のメインコードベース（`src/training/` 配下）に `import optuna` などの探索用ロジックを混入させることを一切禁止する。探索結果は `configs/hparams_<SIZE>.yaml` という不変の静的成果物ファイルとしてGit管理する。

### 3.2 学習フェーズ（オンライン・設定読み取り専用）
- **ポリシー**: 探索結果を合成した設定ファイルを読み込み、即時に学習を起動する。学習プロセスは一切の「探索」「スケーリング計算」などの動的判断を行わない。

## 4. 設定設計ポリシー（責任分離）

構成項目は、環境、人間、および探索ツールの間で関心と責任を厳格に分離する。

### 4.1 責任分離ポリシー
* **環境・コードの自動決定領域**: 
  実行時VRAM検出、混合精度、DeepSpeed ZeROステージなどの環境依存パラメータは、人間が手動指定せず、システムコード（`src/training/config.py` 等）が起動時に自動で解決・設定する。
* **人間の手動指定領域**: 
  モデルサイズ（150M等）、学習制御（エポック数、シークエンス長、データパス）など、実験の予算や前提条件にかかわる項目は人間が YAML 設定を通じて決定する。
* **探索ツールの決定領域**: 
  最適学習率（LR）、Weight Decay（WD）、バッチサイズなどのハイパーパラメータは、人間が手動で調整せず、探索フェーズ（HPO）ツールが理論・実測に基づいて算出し、`hparams_*.yaml` に保存する。

### 4.2 設定合成の単一性ポリシー
- 設定は必ず「共通固定値（`config.yaml`）」に対して「モデルサイズ固有の探索パラメータ（`hparams_XXX.yaml`）」を Hydra の `defaults` 機能で 1 つだけ差し込むパターンに制限する。同種のパラメータを異なる設定ファイル間で重複して定義することを禁止する。

## 5. ハードウェア制約への適応ポリシー

### 5.1 メモリ・VRAMスワップの動的自動回避ポリシー
- Windows WSL2 (Linux) 環境において、低VRAM（≤6GB）の単一GPU環境で実行する際は、OSによるメインメモリ（システムRAM）へのサイレント・スワップ（Paging）による恒久的な減速（`43s/it` などのPCIe転送ボトルネック）を回避するため、以下のポリシーを適用する：
  1. **`torch_compile` の強制無効化**: コンパイル初期のVRAMスパイクによるスワップを防止するため、`torch_compile: false` とする。
  2. **`use_liger_kernel` の優先度低下**: VRAMに余裕がある限り（VRAM消費が2GB以下）、Tritonカスタムカーネルのレジスタ圧迫やコンパイル時グラフブレイクを回避するため `use_liger_kernel: false` とし、最適化されたPyTorch C++/CUDAネイティブカーネルを優先する。

### 5.2 データ配置とI/Oボトルネック排除ポリシー
- WSL2環境下において、`/mnt/c/`（Windowsファイルシステム）のマウント領域またはシンボリックリンク経由でデータセットを読み出すことを完全に禁止する（9PプロトコルによるI/O遅延がGPU待ちを発生させるため）。
- 大容量データセット（`dataset.jsonl`）は、必ずWSL2ローカルの高速な ext4 領域（`~/` 配下）に直接物理ファイルを配置して読み込ませる。

### 5.3 決定論的アルゴリズム（deterministic）の非強制化ポリシー
- 開発および通常の事前学習実行フェーズにおいては、再現性確保のための `deterministic=True` の設定を原則禁止とする。非決定論的アルゴリズムの選択肢を奪うことによる10〜20%の処理速度低下を回避し、ハードウェア性能を最大化する。

### 5.4 手動レジューム（ハッシュ整合性検証）の義務付け
- 学習の意図しない自動起動および不整合状態での起動を防ぐため、自動再開機能は不採用とする。
- 再開（レジューム）は必ず明示的な手動コマンド経由で行い、起動時にチェックポイント保存時の `config.yaml` およびデータセットの SHA-256 ハッシュ値を現在のファイルと検証し、差異がある場合は例外（ValueError）を発生させて処理を中断する。

### 5.5 開発段階の学習量制限ポリシー
- コードおよびパイプラインの動作検証を行う開発段階においては、リソース無駄遣いによる開発サイクルの遅延を防ぐため、`num_epochs` を原則 `1` に制限し、3エポック以上の本番事前学習へのスケールアップはパイプラインの完全な安定稼働が実証された後のみとする。

## 6. 例外的に「書く」ことを許可されるコードポリシー

新規のファイル追加や関数実装は、以下の4条件を**すべて満たす**例外的な場合にのみ許可される。

1. **代替不能性**: 標準ライブラリやライブラリ公式の公開APIで同等の機能が実装されていない。
2. **保守性（サイズ制限）**: 独自実装コードのサイズが100行未満であり、単一責任である。
3. **安定性**: 依存するライブラリの上流アップデートの影響を受けにくい、安定したインターフェースである。
4. **テスト・削除容易性**: 容易にユニットテストが可能であり、不要になった際に依存を引きずらずに削除可能である。

## 7. 変更管理および文書管理ポリシー

### 7.1 文書の最高位決定権
- 本文書は本プロジェクトにおける **最高位決定文書** であり、コードの実装、設定の調整、環境構築の判断において迷いが生じた際の最終決定基準とする。
- ADR (Architecture Decision Records) は本文書の詳細な意思決定理由および変更履歴であり、本文書と矛盾してはならない。本文書の変更を伴う判断を行う際は、必ず新規のADRを策定することによって履歴化する。

### 7.2 GitHub ブランチ保護ポリシー (ADR-0041)
- **Adopt (採用)**:
  - リポジトリの `master` ブランチ（または `main` ブランチ）に対して GitHub のブランチ保護ルール（Branch Protection Rules）を強制適用する。
  - 直接プッシュを全面的に禁止し、すべての変更は必ずプルリクエスト（PR）を経由することを義務付ける。
  - オーナー（`ayato-labs`）の明示的な承認レビュー（Approval）なしでのマージ行為をシステムレベルで完全に禁止・ブロックする。
- **Avoid (不採用)**:
  - リポジトリ管理者による保護ルールのバイパス（管理者による直接プッシュや未承認マージ）を禁止し、管理者自身もルールを適用対象とする（`Do not bypass the above settings` を有効化）。
  - 他ユーザーによる自主的なマージ行為は一切禁止。
