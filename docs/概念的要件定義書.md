# 概念的要件定義書 (ブラッシュアップ版)

## 目的
小説特化のドメイン特化LLMを構築するためのデータ収集および学習用データパイプラインを構築する。学習データを好みの作品だけに限定し、徹底的にノイズを排除することで、極小モデル（40M程度）でも高密度な作風を再現可能な独自LLMの基盤とする。

## 実装方針と技術選定

### 実装するもの・採用する方針
- **データ収集**: `Crawlee` を用いて指定された小説サイトからデータを取得する。
- **データ構造**: SQLite を使用し、サイト別（URL構造の差異）のパーサーを抽象化して実装する。
- **データ管理**: 
    - サイト情報を格納し、識別可能なメタデータ（`site_type`）を付与する。
    - R18の有無を判別するためのフラグ（`is_r18`）を保持する。
- **モデルアーキテクチャ**: 最新の現代的アーキテクチャ（Llamaベース）を採用。GQA (Grouped Query Attention) による推論効率の最大化を図る。
- **長編物語生成**: 物語の整合性を保持するため、物理的なコンテキストウィンドウの拡張のみに依存せず、プロットやキャラ設定を「外部メモリ（RAG/キーバリュー）」として保持し、推論時に動的に注入する階層型推論を採用する。
- **運用UI**: DBeaver を操作UIとして活用。
- **データクレンジング機構**: Web小説特有のノイズを正規表現で自動除去。

### 実装しないもの・あえて排除する方針
- **UI開発**: DBeaverで完結させる。
- **微修正の自動同期**: 本文の微修正は追跡しない。
- **重厚なインフラ**: SQLite（WALモード）によるローカルファイルベースで完結させる。
- **DB内でのトークン化・テンソル化**: Tokenizeは学習時のDataloader側の責務とする。

## LLM構築のための識別要件
本システムで生成するデータセットは、以下のモードのモデル構築を想定する。
1. **ノーマル小説モード**: `is_r18 = 0` 
2. **R18小説生成モード**: `is_r18 = 1` 

モデルの学習効率を最大化するため、DBレベルで「学習に値するクリーンなデータであるか」を明示的にフィルタリング可能な状態を維持する。
