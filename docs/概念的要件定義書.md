# 概念的要件定義書

本書は LLM_Training システムにおいて「何を、何で、なぜその方針で実装するのか」および「何を、なぜ実装しないのか」を定義する。実装の詳細（How）やアーキテクチャ仕様はアーキテクチャ詳細設計書に委譲し、本書では**設計判断の根拠と境界線**のみを扱う。

---

## 1. 本システムの位置づけと責務境界

### 1.1 本システム（LLM_Training）の責務

小説特化のドメイン特化 LLM をフルスクラッチで事前学習（Pre-training）するためのパイプラインである。具体的には以下の3つの工程を担う。

1. **前処理**: 収集済みデータベースからの読み出し、メタデータ算出、学習用 JSONL への変換
2. **学習**: モデルの初期化、ハイパーパラメータ最適化、事前学習の実行
3. **推論**: 学習済みモデルによるテキスト生成

### 1.2 本システムの責務ではないもの

| 責務 | 担当システム |
|------|-------------|
| Web小説の収集・クロール | Novel_Data_Collection |
| サイト別パーサーの実装 | Novel_Data_Collection |
| SQLite スキーマ設計・データクレンジング | Novel_Data_Collection |
| 収集対象作品の選定・管理 | Novel_Data_Collection |
| データベースの操作 UI | DBeaver（外部ツール） |

本システムは Novel_Data_Collection が生成した `novels.db` を**読み取り専用の入力**として受け取る。データの収集・整備に関する方針は Novel_Data_Collection 側の要件定義に従う。

## 2. 設計の前提制約

本システムのすべての設計判断は、以下の3つの制約に支配される。

### 2.1 ハードウェア制約

| 項目 | 値 | 備考 |
|------|-----|------|
| GPU | RTX 3050 Laptop | WDDM ドライバ（Windows 専用） |
| VRAM | 4 GB | 物理上限。超過時は OS レベルでページング発生 |
| CPU RAM | 十分 | ただし学習時の CPU-GPU 間データ転送は致命的ボトルネック |
| OS | Windows | WDDM によるメモリ管理が Linux (TCC) と根本的に異なる |

### 2.2 データ制約

- 学習データは**開発者本人が選定した好みの小説のみ**で構成する
- データ量は数万チャプター規模（数千万トークン級）であり、大規模コーパスではない
- R18 コンテンツと非 R18 コンテンツが混在し、フラグで識別可能

### 2.3 運用制約

- 開発・学習・推論のすべてを Windows ローカル環境で完結させる
- クラウド GPU サービスとの連携は行わない（将来的な GPU レンタルは別途検討）

## 3. 採用する方針とその根拠

### 3.1 モデルアーキテクチャ: Llama ベース

- **採用理由**: Llama アーキテクチャ（GQA, RoPE, SwiGLU, RMSNorm）は、同一パラメータ数において GPT-2 系やオリジナル Transformer に対して一貫して高い性能を示す。極小モデル（120M 規模）では1パラメータあたりの表現力が生命線であり、最も効率的なアーキテクチャを選択する必要がある。
- **棄却した代替案**: GPT-2 アーキテクチャ（MHA + LayerNorm + GELU）。同一パラメータ数での性能で劣後する。

### 3.2 モデルサイズ: 120M パラメータ

- **採用理由**: VRAM 4GB 環境で GPU メモリ内に静的重量・勾配・オプティマイザ状態・活性化メモリが 100% 収まる最大規模。FP16 での推定メモリ使用量は約 0.39GB（モデルのみ）、学習時の総消費は約 1.6GB 以内に収まる。
- **棄却した代替案**:
  - 40M-100M（過小）: パラメータ不足により小説ドメインの表現力が不足する
  - 150M 超（過大）: 学習時にページング閾値を超えるリスクがある

### 3.3 スタイル条件付き学習（Conditioned Pre-training）

- **採用理由**: 極小モデルでは暗黙的にスタイルを学習する容量が不足する。データセットのエクスポート時に各チャンクの冒頭へ「作品名」「ジャンル」「タグ」「会話率」「感情」「文体比率」をメタデータプレフィックスとして結合することで、推論時にメタデータを指定するだけでスタイルを陽に制御できる（ステアリング能力）。
- **棄却した代替案**: ファインチューニングによるスタイル制御。追加の学習フェーズが必要であり、データ量が限定的な本プロジェクトでは過学習のリスクが高い。

### 3.4 スケール則による自動ハイパーパラメータ最適化

- **採用理由**: Step Law（arXiv:2503.04715）を用いて、パラメータ数とトークン数から最適な学習率・バッチサイズを理論的に算出する。4GB VRAM 環境では大量のグリッドサーチを実行する余裕がないため、理論値による一発計算が不可欠。
- **補完手段**: 小規模な代理モデル（Proxy、ターゲットの約 25% 規模、最低 30M）で少数の学習率候補を短時間探索し、最良の学習率をターゲットモデルに外挿する。代理モデルの下限を 30M に設定するのは、それ以下ではスケール則の予測精度が著しく劣化するため。
- **棄却した代替案**: フルスケールでのグリッドサーチ。120M モデルでの試行1回あたり数時間を要し、4GB VRAM 環境では現実的でない。

### 3.5 分離最適化（Muon + AdamW）

- **採用理由**: 2D パラメータ（重み行列）に Muon、1D パラメータ（バイアス、LayerNorm、Embedding 等）に AdamW を振り分ける。Muon は大規模な重み行列に対してスケーラブルな収束性能を示し、将来的なモデルスケールアップ時にも学習率の再調整が最小限で済む。
- **フォールバック**: Muon がインストールされていない環境では、2D/1D 双方に AdamW を異なる学習率で適用する。

### 3.6 GPU オンリー学習の強制

- **採用理由**: 実地計測により、CPU オフロード使用時のステップ所要時間が約 15 倍（約 3 秒 → 約 45 秒/step）に劣化することを確認した。原因は PCIe バス経由のパラメータ同期である。4GB VRAM に収まるモデルサイズ（120M）に厳格に制限することで、CPU オフロードを完全に不要とする。
- **棄却した代替案**: DeepSpeed ZeRO-Offload によるCPUオフロード。理論上はより大きなモデルを学習可能だが、速度劣化が致命的。

### 3.7 VRAM 連動の静的バッチサイズ制限

- **採用理由**: Windows の WDDM メモリ管理下では、物理 VRAM を超過した瞬間に OS レベルのページング（メインメモリへの退避・復帰）が発生し、ステップ所要時間が 10 分以上に膨れ上がる。この閾値超過を防ぐため、物理 VRAM 容量に応じたバッチサイズの上限を静的に設定する（4GB 環境では最大 2）。
- **棄却した代替案**: `auto_find_batch_size=True`（HuggingFace Trainer の自動バッチサイズ探索）。OOM 発生時にバッチサイズを半減して全データセットを最初から再ロード・再トークナイズする仕組みであり、リトライのたびに数十分のオーバーヘッドが積み上がる。実測で step 2 の所要時間が 1342 秒（通常の 15 倍）に膨れ上がることを確認した。

### 3.8 DeepSpeed ZeRO Stage 2（GPU オンリー構成）

- **採用理由**: 単一 GPU 環境でも、勾配の allgather/reduce_scatter の最適化と通信のオーバーラップにより、素の PyTorch DDP より効率的なメモリ利用と若干のスループット向上が得られる。CPU オフロードは無効化した状態で使用する。
- **棄却した代替案**: DeepSpeed なし（素の Trainer）。Gradient Checkpointing との組み合わせで動作するが、メモリ効率で劣る。

### 3.9 混合精度学習（FP16）

- **採用理由**: VRAM 消費をほぼ半減でき、4GB 制約下では必須の選択。RTX 3050 は FP16 の Tensor Core を搭載しており、計算スループットも向上する。
- **棄却した代替案**: BF16（RTX 3050 は BF16 の Tensor Core 非搭載）、FP32（VRAM 消費が 2 倍）。

### 3.10 勾配チェックポインティング（Gradient Checkpointing）

- **採用理由**: フォワードパスの中間活性化を保持せず逆伝播時に再計算することで、活性化メモリを大幅に削減する。4GB 環境でバッチサイズ 2 を確保するために必須。計算時間は約 20% 増加するが、メモリ削減効果の方が圧倒的に重要。

### 3.11 オーバーラップ付きテキスト分割

- **採用理由**: 小説の章テキストはコンテキストウィンドウ長（1024 トークン、約 2000 文字）を超えることが多い。文脈の断絶を最小化するため、チャンク間に 200 文字のオーバーラップを設けて分割する。
- **棄却した代替案**: 単純な切り捨て（文脈が喪失する）、Token Packing（将来検討。現在のデータ規模では実装コストに見合う効果が限定的）。

### 3.12 階層型推論（RAG ベース）

- **採用理由**: 120M モデルのコンテキストウィンドウ（1024 トークン）では長編小説の文脈を保持できない。プロット・キャラクター設定・過去の要約を外部メモリとして保持し、推論時に動的にプロンプトへ注入することで、物理的な制約を超えた整合性維持を実現する。
- **棄却した代替案**: コンテキストウィンドウの拡大（4096 等）。VRAM 消費が二次関数的に増大し、4GB 環境では不可能。

### 3.13 R18/ノーマルのモード分離

- **採用理由**: `is_r18` フラグにより学習データをフィルタリング可能とし、用途に応じたモデルを構築できる。R18 コンテンツを含むモデルと含まないモデルを独立に管理する。

## 4. 採用しない方針とその根拠

### 4.1 外部評価情報の学習への利用

- **除外対象**: ブックマーク数、星の数、ランキング順位等の読者評価メトリクス
- **除外理由**: 本システムは「多数派の好み」ではなく「開発者個人の好みの作風」を再現することが目的。外部評価は多数派のバイアスを注入し、ターゲットとする作風の純度を毀損する。
- **再検討条件**: なし。これはシステムの根幹的な設計思想であり、変更は目的の変更を意味する。

### 4.2 Google Colab / クラウド GPU 連携

- **除外対象**: Colab 向けの環境差異吸収ロジック、リモート同期機構
- **除外理由**: パス構造、改行コード、ランタイム制限（Colab は最大 12 時間）等の環境差異を吸収する開発・保守コストが、得られる計算リソースに見合わない。
- **再検討条件**: ローカルの `.py` ロジックをそのまま Colab 上で実行できる画期的な OSS が登場した場合。または、GPU レンタルサービス（RunPod, Vast.ai 等）を利用し Remote-SSH で開発する場合は本制約の対象外。

### 4.3 専用 UI の開発

- **除外対象**: Web UI、デスクトップアプリケーション等の独自 UI
- **除外理由**: データの閲覧・管理は DBeaver で十分に完結する。推論 UI は将来必要になるが、現フェーズ（学習基盤の確立）では優先度が低い。
- **再検討条件**: 学習済みモデルの品質が十分に高くなり、対話的な推論・評価が日常的に必要になった場合。

### 4.4 DB 内でのトークン化・テンソル化

- **除外対象**: SQLite 内にトークン化済みデータやテンソルを格納する方式
- **除外理由**: トークナイザーの変更やシーケンス長の変更のたびに DB を再構築する必要が生じ、柔軟性を損なう。トークン化は学習時の DataLoader 側の責務とする。
- **再検討条件**: データ規模が 100 億トークンを超え、毎回のトークン化処理がボトルネックになった場合。

### 4.5 本文微修正の自動追跡

- **除外対象**: 収集済み小説の本文が更新された場合の差分検出・自動更新
- **除外理由**: 実装コストに対して効果が薄い。必要な場合は手動で再収集する。
- **再検討条件**: なし。

### 4.6 Token Packing（Sequence Concatenation）

- **除外対象**: 複数ドキュメントを EOS トークンで区切って1シーケンスに敷き詰める手法
- **除外理由**: 現在のデータ規模（約 3000 万トークン）では、`padding="max_length"` による PAD トークンの計算コストは、実装の複雑化に対して許容範囲内。
- **再検討条件**: データ規模が数十億トークンにスケールし、PAD トークンの計算コストが無視できなくなった場合。

### 4.7 重厚なインフラストラクチャ

- **除外対象**: PostgreSQL、分散ファイルシステム、コンテナオーケストレーション等
- **除外理由**: 単一マシン・単一 GPU でのローカル実行が前提。SQLite（WAL モード）で読み取り性能は十分であり、運用の複雑性を増す理由がない。
- **再検討条件**: 複数マシンでの分散学習が必要になった場合。

## 5. 実地検証から得た制約（経験的教訓）

以下は実際の学習実行から得られた教訓であり、理論的な推測ではなく計測結果に基づく。

### 5.1 CPU オフロードの禁止

| 項目 | 内容 |
|------|------|
| **現象** | 120M モデルの学習ステップが約 3 秒/step から約 45 秒/step に劣化（約 15 倍） |
| **原因** | DeepSpeed ZeRO-Offload が毎ステップでパラメータ・勾配を PCIe バス経由で CPU ⇔ GPU 間転送。RTX 3050 Laptop の PCIe 帯域（Gen4 x8）がボトルネック |
| **対策** | `generate_deepspeed_config()` で CPU オフロードを常時無効化。モデルサイズを GPU メモリ内に収まる 120M に制限 |
| **分類** | **非機能要件（絶対制約）** — 事前学習においては例外なく適用 |

### 5.2 auto_find_batch_size の禁止

| 項目 | 内容 |
|------|------|
| **現象** | step 1 が 82 秒で完了後、step 2 が 1342 秒（約 16 倍）に膨張 |
| **原因** | `auto_find_batch_size=True` により、OOM 発生時に HuggingFace Trainer がバッチサイズを半減し、データセット全体（約 36,000 件）を最初から再ロード・再トークナイズしてリトライ。OOM が断続的に発生するたびにこの再処理コストが積み上がる |
| **対策** | `auto_find_batch_size` を削除し、VRAM 容量に基づく静的なバッチサイズ上限を設定（4GB 環境では最大 2） |
| **分類** | **非機能要件（絶対制約）** — HuggingFace Trainer の当該オプションは使用禁止 |

### 5.3 WDDM ページングの回避

| 項目 | 内容 |
|------|------|
| **現象** | VRAM 使用量が物理上限（4GB）を超過した際、ステップ所要時間が 10 分以上に急増 |
| **原因** | Windows の WDDM ドライバが共有メモリ（メインメモリ）へ物理ページを退避・復帰させる OS レベルのオーバーヘッド。Linux の TCC ドライバとは根本的にメモリ管理方式が異なる |
| **対策** | バッチサイズの静的上限を物理 VRAM に連動させ、ページング閾値を超過しないことを保証する |
| **分類** | **非機能要件（環境依存制約）** — Windows + WDDM 環境に固有 |

## 6. 品質基準

### 6.1 学習データの品質基準

- **厳選原則**: 開発者本人が読了し、作風・文体を再現する価値があると判断した作品のみを学習データとする
- **ノイズ排除**: 前書き・後書き・作者コメント・ルビ等の Web 小説特有のノイズは前処理段階で除去済みであること（Novel_Data_Collection 側の責務だが、本システムはクリーンなデータが入力される前提で設計する）
- **メタデータ整合性**: 各チャンクに付与されるメタデータプレフィックス（会話率、感情、文体比率等）が本文の実態と一致していること

### 6.2 モデルの成功基準

1. **学習 Loss の収束**: 学習曲線が右肩下がりの収束傾向を示すこと
2. **作風の再現性**: 出力テキストに、学習データに含まれる特定の作品の語彙選択・セリフ回し・描写のテンポの片鱗が観測されること

この 2 点が確認できた場合、本アーキテクチャは有効であると判断し、より大規模な計算リソースでのスケールアップを検討する。
