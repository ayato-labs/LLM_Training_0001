# 概念的要件定義書 (ブラッシュアップ版)

## 目的
小説特化のドメイン特化LLMを構築するためのデータ収集および学習用データパイプラインを構築する。学習データを好みの作品だけに限定し、徹底的にノイズを排除することで、極小モデル（150M程度）でも高密度な作風を再現可能な独自LLMの基盤とする。

このシステムの役割は、"C:\Users\saiha\My_Service\programing\LLM\Novel_LLM\Novel_Data_Collection"で集めたデータを使って学習することであり、データ収集は"C:\Users\saiha\My_Service\programing\LLM\Novel_LLM\Novel_Data_Collection"の役割で、このシステムの役割ではないです。

## 実装方針と技術選定

### 実装するもの・採用する方針
- **データ収集**: `Crawlee` を用いて指定された小説サイトからデータを取得する。
- **データ構造**: SQLite を使用し、サイト別（URL構造の差異）のパーサーを抽象化して実装する。
- **データ管理**: 
    - サイト情報を格納し、識別可能なメタデータ（`site_type`）を付与する。
    - R18の有無を判別するためのフラグ（`is_r18`）を保持する。
- **モデルアーキテクチャ**: 最新の現代的アーキテクチャ（Llamaベース）を採用。GQA (Grouped Query Attention) による推論効率の最大化を図る。
- **長編物語生成**: 物語の整合性を保持するため、物理的なコンテキストウィンドウの拡張のみに依存せず、プロットやキャラ設定を「外部メモリ（RAG/キーバリュー）」として保持し、推論時に動的に注入する階層型推論を採用する。
- **運用UI**: DBeaver を操作UIとして活用。
- **データクレンジング機構**: Web小説特有のノイズを正規表現で自動除去。
- **スケーラブルな学習インフラ**:
    - **動的VRAM最適化**: 実行環境のGPU物理メモリ総容量を PyTorch から自動検出。メモリ容量に基づいて DeepSpeed CPU オフロード（Stage 2）の有効/無効を動的に切り替え、スループットを最大化する。
    - **Muon + AdamW 分離最適化**: 最新の最適化スケーリングアルゴリズムを採用。2Dパラメータ（重み行列）に Muon、1Dパラメータ（バイアス、レイヤーノーマライゼーション、埋め込み等）に AdamW を振り分け、大規模パラメータへのスケールアップ時にも高速な収束を実現する。

### 実装しないもの・あえて排除する方針
- **外部評価情報（読者評価やブクマ数等）の学習からの除外**: データベース側に蓄積されている場合でも、学習対象データを「開発者本人の好みの小説（厳選データ）」だけに限定してアライメントを行うため、多数派の指標である外部評価情報（ランキング、ブクマ数、星の数等）は学習時の入力メタデータ（プレフィックス）としては利用せず、意図的に除外する。
- **UI開発**: DBeaverで完結させる。
- **微修正の自動同期**: 本文の微修正は追跡しない。
- **重厚なインフラ**: SQLite（WALモード）によるローカルファイルベースで完結させる。
- **DB内でのトークン化・テンソル化**: Tokenizeは学習時のDataloader側の責務とする。
- **Google Colab向け環境の構築**: Google Colab の計算リソースを活用するための複雑な連携・変換ロジックの構築は行わない。環境差異（パスや改行コード）の吸収や同期などの開発・保守コストが非常に大きいため、必要に応じて専用の GPU 仮想マシン（RunPod や Vast.ai 等）をレンタルして Remote-SSH で開発する。ただし、ローカルの `.py` ロジックをそのまま簡単に Colab 上で実行できるような画期的なOSSが登場した場合は再検討する。

## LLM構築のための識別要件
本システムで生成するデータセットは、以下のモードのモデル構築を想定する。
1. **ノーマル小説モード**: `is_r18 = 0` 
2. **R18小説生成モード**: `is_r18 = 1` 

モデルの学習効率を最大化するため、DBレベルで「学習に値するクリーンなデータであるか」を明示的にフィルタリング可能な状態を維持する。
