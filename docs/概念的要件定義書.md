## 1. 根本原則：引き算のエンジニアリング

> **「コードを書くことは借金である。エンジニアリングは引き算をする職人仕事である。」**

本プロジェクトにおける全ての技術的判断は、以下の優先順位で行われる。

1. **書かない**（標準ライブラリ・上流フレームワークに委譲）
2. **削除する**（独自実装・デッドコード・重複機能）
3. **分離する**（実行時計算 → 事前計算アーティファクト）
4. **統一する**（二重管理・複数依存の単一化）
5. **やむを得ない時だけ書く**（制約上不可避な最小限の接着コード）

**実装しないことの定義こそが、実装することの定義より重要である。**

### 1.1 開発ロードマップ戦略：既存ライブラリ主導のプロトタイプ構築

1. **第1段階：既存ライブラリを組み合わせたプロトタイプ構築**
   - まずは既存ライブラリを活用し、「チンチラ則×スケール則×小規模モデル（プロキシモデル）」を用いたハイパーパラメータの探索ロジックを導入した、効率的なLLM学習アーキテクチャを組み上げる。
   - 最初のプロトタイプシステムは、完全に既存ライブラリの組み合わせによって動作し、信頼できる探索パイプラインを最速で構築することを最優先とする。
2. **第2段階：ライブラリで賄えない部分への着手**
   - 既存ライブラリによるプロトタイプアーキテクチャが完全に組み上がり、安定稼働したことを確認した後に、標準ライブラリやフレームワークではどうしても実現できない独自ロジックやカスタム部分の最適化・開発に着手する。

## 2. アーキテクチャ憲章

### 2.1 採用する標準スタック（委譲先）

| 領域 | 採用 | 理由 |
|---|---|---|
| **モデル実装** | `transformers.LlamaForCausalLM` | Flash Attention 2、ZeRO/FSDP/TPネイティブ、カーネル融合、上流バグ修正自動享受 |
| **学習ループ** | `transformers.Trainer` | 勾配累積・AMP・チェックポインティング・分散・スケジューラ統合が成熟 |
| **オプティマイザ** | `AdamW` (PyTorch標準) | ZeRO最適化との親和性最大、実装成熟、ハイパラ知見豊富 |
| **設定管理** | `Hydra` + `OmegaConf` | 合成・オーバーライド・型検証・CLI標準化 |
| **ログ基盤** | `TensorBoard` のみ | ローカル完結・軽量・VSCodeネイティブ・依存最小 |
| **データ処理** | `datasets` + `tokenizers` (Rust) | ストリーミング・並列化・メモリ効率 |
| **分散・メモリ** | `DeepSpeed` (ZeRO-1/2/3 + Offload) | VRAM自動適応、HF Trainer統合済み |
| **環境管理** | `uv` + `.venv` | 高速・ロックファイル・キャッシュ・Windows対応 |

### 2.2 明示的に採用しないもの（アンチゴール）

| 領域 | 不採用 | 理由 |
|---|---|---|
| **独自モデル実装** | `ModernGPT`, `CustomTransformer` 等 | 標準実装で十分。独自カーネル保守は借金。 |
| **独自Trainer** | `CustomTrainer`, 手動学習ループ | `Trainer` で全要件充足。バグ温床化。 |
| **Muon / 独自オプティマイザ** | `normuon.py`, 2D直交化更新 | AdamW+ZeROで代替可能。LRスケール調整で吸収。 |
| **MLflow** | 実験管理・モデルレジストリ | TensorBoardで十分。依存・DB・UI起動コストが借金。 |
| **Optuna (学習時)** | 学習起動時のHPO実行 | リードタイム3-5分は無駄。オフラインアーティファクト化で解決。 |
| **複数設定ファイル** | `scaling_*.yaml` 乱立 | Hydra `defaults` + 単一 `hparams.yaml` で単一化。 |
| **HF Hub経由Tokenizer** | `from_pretrained("path/to/tokenizer.json")` | ネットワーク・認証・レートリミット依存。ローカル直接読み込みで解決。 |
| **デーモンプロセス** | 別プロセスDriveアップローダー | `TrainerCallback` で同期統合。プロセス管理不要。 |
| **手動VRAM/DeepSpeed設定** | YAMLハードコード | `torch.cuda.get_device_properties` 自動検出 + ZeROステージ自動選択。 |

## 3. フェーズ分離アーキテクチャ

### 3.1 探索フェーズ（オフライン・非同期・成果物指向）

```
入力: モデルサイズ、データ量、VRAM、シーケンス長
処理: Step Law (Chinchilla/μP) Prior → Optuna Proxy探索 (TPE multivariate + MedianPruner)
成果物: configs/hparams_<SIZE>.yaml (Git管理)
頻度: モデル/データ/VRAM変更時のみ（数時間）
実行: `run_hpo.bat` / `scripts/find_hparams.py`
依存: optuna (dev依存のみ)
```

**方針**: 学習パイプラインから**完全に分離**する。学習コードに `import optuna` を書かせない。
MLflowは探索フェーズでは使用しない（TensorBoardのみ）。

**方針**: 学習パイプラインから**完全に分離**する。学習コードに `import optuna` を書かせない。

### 3.2 学習フェーズ（オンライン・即時開始・設定読み取り専用）

```
入力: config.yaml + hparams_<SIZE>.yaml (Hydra合成)
処理: HF Trainer 即時実行
出力: models/output/ (checkpoint, tensorboard, final model)
頻度: 開発サイクル毎（即時）
実行: `run_train.bat` / `python -m src.main`
依存: torch, transformers, datasets, accelerate, deepspeed, tensorboard のみ
```

**方針**: 起動から学習開始まで **10秒以内**。探索・計算・判断を含まない。

## 4. 設定設計原則

### 4.1 責任分離マトリクス

| 設定項目 | 定義場所 | 変更主体 | 変更頻度 | 理由 |
|---|---|---|---|---|
| VRAM検出・Precision・ZeRO Stage | **コード** (`config.py`, `model_utils.py`) | システム/コード | ほぼ無し | 環境依存・人間が決められない |
| モデルアーキテクチャ (層数/ヘッド/隠れ次元) | **config.yaml** | 人間 | 実験毎 | 意図的なアーキテクチャ選択 |
| 最適ハイパラ (LR/WD/Batch/Clip) | **hparams_XXX.yaml** | **探索ツール** | モデル/データ/VRAM変更時 | 理論+実証で決まる。人間が手調整しない。 |
| 学習制御 (steps/epoch/seq_len) | **config.yaml** | 人間 | 実験毎 | 予算・データ量による判断 |
| データパス・Tokenizerパス | **config.yaml** | 人間 | 環境構築時 | 物理配置依存 |

### 4.2 Hydra `defaults` パターンの強制

```yaml
# config.yaml
defaults:
  - hparams_150M   # 必ず単一のhparamsファイルを合成
# _self_ 暗黙適用でベース設定が後勝ち上書きされる
```

- `scaling_*.yaml` のような「ベースコピー＋差分」パターンを**禁止**
- モデルサイズ変更 = `hparams_3B.yaml` 作成 → `defaults: [hparams_3B]` 書き換えのみ

## 5. ハードウェア制約への適応戦略

### 5.1 単一GPU・VRAM 4GB〜24GB 完全自動対応

| VRAM帯域 | ZeRO Stage | Offload | 期待動作 |
|---|---|---|---|
| ≤ 5GB | Stage 3 | Param + Optimizer → CPU | 学習可能（低速許容） |
| 6〜10GB | Stage 2 | なし | 標準的学習 |
| 11〜20GB | Stage 1/2 | なし | 高速学習 |
| ≥ 21GB | Stage 1 | なし | 最大バッチ・高速 |

**実装**: `model_utils.generate_deepspeed_config(n_params, vram_gb)` で自動生成。YAML設定不要。

### 5.2 ストレージ制約（ローカルSSD逼迫）

- チェックポイントは **ローカルに最新N個のみ保持**（`save_total_limit=2` による自動クリーンアップ）
- 必要に応じて、非同期で `scripts/drive_uploader.py` を実行し Google Drive へバックアップ可能
- DVC・MLflowアーティファクトストア **不使用**（依存・容量・複雑性の観点から）

### 5.3 学習中断からの再開（レジューム）設計

- **明示的な手動再開（コマンド）の採用**:
  - 設定条件の一致による自動再開機能は**不採用**とする。
  - 意図しない自動起動を防ぎ、再開処理の安全性・確実性を最優先するため、`--resume` 引数を付与した明示的な手動コマンド（`run_train.bat --resume`）によってのみ再開する。
- **再開時の厳格なハッシュ検証**:
  - 意図しない不整合（設定の変更や学習データの差し替え）を防ぐため、チェックポイント保存時の `config.yaml` および学習データファイルの SHA-256 ハッシュ値を現在のファイルと検証する。
  - ハッシュ値に差分がある場合は例外（ValueError）を発生させ、不適切な状態での再開を未然に防止する。
- **自動再開の検討条件（将来的な拡張ロードマップ）**:
  - 自動起動・自動再開の導入検討は、以下のいずれかの条件を満たす場合に限定する。
    1. 学習環境がクラウド（スポットインスタンス等のプリエンプティブな環境）へ移行し、ノードの自動再起動が必須要件となった場合
    2. 現在の手動レジュームロジックが完全に完成・安定稼働し、その上の「余録（追加機能）」として実装する十分な余力がある場合
  - 現段階のローカル環境においては、複雑性を持ち込まず、予測可能性の最も高い「手動コマンドでの再開」を堅持する。

## 6. 再現性・トレーサビリティ

### 6.1 学習実行時に自動記録される最小メタデータ

`src/env_snapshot.py` が `Trainer` 起動時に `models/output/runs/environment.json` へ出力：

```json
{
  "git_hash": "abc123...",
  "git_dirty": false,
  "python": "3.12.10",
  "torch": "2.13.0",
  "cuda": "12.1",
  "gpu": "RTX 4090",
  "transformers": "4.40.0",
  "datasets": "2.19.0",
  "deepspeed": "0.14.0"
}
```

### 6.2 設定の完全固定

- `hparams_XXX.yaml` は **Gitコミット対象**（探索結果の不変アーティファクト）
- `config.yaml` は **手動編集履歴としてGit管理**
- Hydra合成済み設定は `Trainer` ログディレクトリに `config.yaml` として自動保存

## 7. 運用ワークフロー（標準手順）

### 7.1 初回・構造変更時

```bat
run_hpo.bat 150M data/dataset.jsonl configs/hparams_150M.yaml 20 4 1024
# 数時間待機 → hparams_150M.yaml 生成 → Git commit
```

### 7.2 日常開発・実験

```bat
run_train.bat                    # 本番フル学習
run_train.bat --max-steps 100    # デバッグ短縮
run_train.bat --resume           # 中断再開
```

### 7.3 スケール変更 (150M → 3B 等)

1. `config.yaml` の `model.target_params` / `model.llama.*` 書き換え
2. `run_hpo.bat 3B ...` で `hparams_3B.yaml` 生成
3. `config.yaml` の `defaults: [hparams_3B]` 書き換え
4. `run_train.bat`

## 8. 例外的に「書く」ことを許可されるコード

以下の条件を**全て満たす**場合のみ、新規ファイル・関数追加を許可：

1. **標準ライブラリ・HF公式APIで代替不能** であること
2. **100行未満** で単一責任であること
3. **インターフェースが安定** し、上流変更に追従不要であること
4. **テスト可能** であり、削除容易であること

現在許可されている独自実装：
- `scripts/drive_uploader.py` - GDrive APIラッパー（バッチ実行用外部ツール）
- `src/step_law.py` - Step Law数式実装（論文固有、標準化されていない）
- `src/config.py` - Hydra正規化ロジック（プロジェクト固有スキーマ）
- `src/env_snapshot.py` - 環境メタデータ収集（標準なし）
- `run_*.bat` - Windowsランチャー（`uv` + Hydra CLIラップ）

## 9. 変更管理ルール

| 変更種別 | 承認要件 | 更新対象 |
|---|---|---|
| **新規依存追加** (本番) | 原則不可。`dev`依存へ逃せないか検討必須 | `pyproject.toml` |
| **独自実装追加** | 本書 §8 の4条件全合致 + ADR新規作成 | `src/*.py` 新規 |
| **設定項目追加** | コード側（自動）かYAML側（人間）か責任分離表に従う | `config.yaml` / `hparams_*.yaml` / コード |
| **標準ライブラリ置換** | 既存独自実装の削減が目的の場合のみ可 | 該当ファイル削除 + 委譲コード |
